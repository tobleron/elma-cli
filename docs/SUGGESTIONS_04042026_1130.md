# Feature Suggestions for Elma CLI

This report outlines high-impact, strategic feature recommendations for Elma CLI based on architectural observation of reference agent implementations (Claude Code, OpenCode). These recommendations avoid existing tasks in `_tasks/TASKS.md`.

## I. Strategic Importance (Architectural Alignment)

**A. Unified Context Observability Service**
- **Gap**: Reference codebases often implement deep, structured context-tracking for agent memory that persists beyond the current turn.
- **Recommendation**: Implement a dedicated `ContextObserver` agent skill/module. This module would not just store messages, but actively summarize session goals, identify drift between "current state" and "intended state," and provide periodic "reflection checkpoints" for the orchestrator.
- **Why**: Enhances Elma's long-term task coherence.

**B. Virtual Tool/Capability Sandbox**
- **Gap**: Reference agents often leverage isolated execution environments that go beyond simple shell commands, supporting complex, multi-modal, or stateful tool environments.
- **Recommendation**: Create a "Virtual Workspace" provider interface. This layer would allow Elma to execute tools within a controlled, state-preserving memory environment (similar to how some agents persist shell states).
- **Why**: Essential for robust, repeatable agent operations that are isolated from the host environment.

## II. Maximum Gain / Minimal Effort

**A. Diagnostic-Trace Tooling**
- **Gap**: Elma lacks a simple command to output the current "reasoning trace" or "thought dependency graph."
- **Recommendation**: Add a `/trace` command that renders the current OODA loop dependency graph for a pending or active task.
- **Why**: Dramatically improves user ability to debug *why* Elma chose a specific path.
- **Effort**: Low (Leverages existing planning data).

**B. Semantic "Tool Search" Pre-flight**
- **Gap**: Reference tools often perform a quick semantic scan of the environment before tool execution to suggest missing capabilities.
- **Recommendation**: Add an optional `PreFlightCapabilityCheck` intel unit that triggers *before* a tool is executed if the environment is ambiguous.
- **Why**: Reduces retry loops caused by missing dependencies or incorrect paths.
- **Effort**: Medium (Can reuse existing intel unit infrastructure).

**C. "Dry Run" Execution Mode**
- **Gap**: Elma usually executes actions directly.
- **Recommendation**: Add a global `DryRun` flag that forces the agent to formulate plans and inspect tool signatures without executing them.
- **Why**: Provides a "safety valve" for complex operations in sensitive repositories.
- **Effort**: Low.
