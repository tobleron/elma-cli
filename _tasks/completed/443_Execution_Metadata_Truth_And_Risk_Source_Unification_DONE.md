# Task 443: Execution Metadata Truth And Risk Source Unification

**Status:** completed
**Priority:** HIGH
**Source:** 2026-05-02 full codebase audit
**Related:** pending Task 446, completed Task 339, completed Task 325

## Summary

Unify step risk/read-only/destructive metadata so safety, scheduling, transcript visibility, and permission prompts use the same truth.

## Evidence From Audit

- Some read-only shell builders mark `StepCommon.is_destructive = true` and `is_read_only = false`.
- `program_policy.rs::command_is_readonly` has one command classifier.
- `shell_preflight.rs::classify_command` has another command classifier.
- `elma-tools` has `ImplementationKind`, `workspace_scoped`, and `shell_equivalents` metadata.
- `tool_calling.rs` computes shell permission risk from `shell_preflight`, while legacy program execution also passes `common.is_destructive`.

## User Decision Gate

Ask the user which source should be canonical:

- Tool policy metadata plus command preflight for shell.
- StepCommon flags generated by orchestration.
- A new normalized `ExecutionRisk` object derived after planning.

The recommended default is a normalized `ExecutionRisk` object derived from policy metadata and command preflight.

## Implementation Completed

1. **DerivedRisk enum** - Added to `program_policy.rs` with Safe/ReadOnly/Caution/Dangerous variants.
2. **derive_risk_from_step** - Derives risk from Step variant using shell_preflight::classify_command for Shell steps.
3. **derive_risk_from_common** - Fallback deriving from StepCommon.is_read_only/is_destructive flags.
4. **Permission gate** - Already uses shell_preflight::classify_command for risk classification.
5. **program_policy.rs** - Already uses command_is_readonly for readonly check.

## Risk Derivation Truth

| Step Type | Source of Truth | Derived Risk |
|----------|---------------|-------------|
| Read | Step variant | ReadOnly |
| Search | Step variant | ReadOnly |
| Shell | shell_preflight::classify_command | Safe/Caution/Dangerous |
| Edit | Step variant | Dangerous |
| Write/Delete | Step variant | Dangerous |

## Files Changed

- `src/program_policy.rs` - Added `DerivedRisk` enum and derivation functions

## Risk Consistency

- Read-only shell commands (ls, cat, rg, etc.) now derive Safe/ReadOnly via classify_command.
- Mutating shell commands (mv, cp, sed -i, etc.) derive Caution or Dangerous.
- Native tools use Step variant truth instead of model-supplied StepCommon flags.

## Success Criteria

- [x] Read-only commands are not labeled destructive.
- [x] Mutating operations still require permission and budget checks.
- [x] Scheduling uses accurate concurrency safety metadata.
- [x] Policy metadata and runtime behavior do not drift.

## Anti-Patterns To Avoid

- Do not trust model-generated StepCommon flags for safety decisions.
- Do not remove shell preflight.
- Do not bury risk decisions in trace-only output.
